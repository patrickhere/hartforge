cutting over a branch

july 2026 · what a weekend cutover actually takes

a community bank branch cutover is a migration with a deadline you don't get to move: the teller line has to be live when the doors open. you get a window, usually a weekend or an after-hours block, and inside it you replace or rebuild the network, the servers, and every workstation, then prove all of it works before staff walk in. no partial credit. if a teller can't pull up a customer at 9am, the branch calls it a failed cutover no matter how clean the rack looks.

that constraint is the whole job. everything below is built around it.

before the window

the cutover is mostly won or lost before i show up.

  • inventory first. every workstation, printer, phone, switch, and the core banking endpoints. i want a list of what's there and what it talks to, because the thing that breaks at 8:45am is always the one device nobody mentioned.
  • configs staged ahead. switch configs written and reviewed, DHCP scopes sized, GPOs confirmed applying in a test OU, profiles and images prepped. i'm not writing a VLAN config at midnight.
  • comms with the branch manager. one contact, a confirmed window, and confirmation that i can actually take the line down. banks have compliance and after-hours access rules, and learning them at the door costs an hour you didn't budget.
  • rollback defined in writing. what abort looks like, the time i pull the trigger, and how the old setup comes back. if there's no rollback there's no cutover, there's a gamble.

the window

order matters. i work bottom up, because nothing above the network works until the network works.

network. switch in, uplinks verified, access ports and the voice VLAN configured before anything plugs in. i bring ports up in a controlled order, because a branch patch panel is exactly where a layer 2 loop wants to happen, and an err-disabled port at hour one is a lot cheaper than one at hour six.

servers and domain. domain reachable, AD and DNS answering, GPO applying to the branch OU. i confirm that with a real client before i touch a single workstation, because a workstation that joins a half-configured domain is a workstation i get to fix twice.

workstations. transwiz for the profile and data migration, then domain join, then confirm the GPO landed. one machine end to end first as the canary, then the rest once the canary is clean.

endpoints and peripherals. new endpoint agent installed, the old one removed cleanly. SEP leaves ghosts if you let it, so that's a silent uninstall, not a right-click. printers mapped, phones up on the voice VLAN.

where it goes sideways

it always does somewhere. the value isn't that nothing breaks, it's that the breakage is boring and i've seen it before.

  • port goes err-disabled. usually a loop or a duplicate, occasionally BPDU guard doing its job. i trace the MAC to the port, find the offending cable or device, clear it, and bring the port back up deliberately. i don't just no shut and pray.
  • workstation pulls the wrong IP, or none. DHCP scope exhausted, a stale lease, or the machine landed on the wrong VLAN. a lease lookup on the windows DHCP server tells me which of the three it is in about a minute.
  • GPO not applying. wrong OU, replication lag, or a filter i missed. gpresult on the canary before i roll the fleet, every time, because finding it on machine one is a fix and finding it on machine twenty is a night.
  • phones dead. almost always the voice VLAN, either the port config or the DHCP option. this is exactly why phones come after the network is proven, not during it.

validation

the branch isn't cut over until it's proven, not until it's plugged in.

  • a real test transaction through the core banking app, not "the app opens."
  • every teller station logs in the way a teller would and pulls a customer.
  • phones dial out and receive.
  • printers print from the machines that need them, including the receipt and check printers, which are always the ones nobody tested.

i walk the line as if i were the staff, station by station. the sign-off is "monday morning works," nothing less.

what i standardized

i stopped running cutovers as one-offs early. after the first few, the process itself became the deliverable:

  • a per-branch checklist that encodes the order above, so the sequence doesn't live in my head.
  • a network baseline, so every branch switch config starts from the same known-good template instead of a blank prompt.
  • a documented transwiz and domain-join flow, so the workstation step is repeatable by whoever is onsite, not just me.

the goal was never to be the person who can cut over a branch. it was to make cutting over a branch a repeatable procedure that survives me not being in the room. that's the part i'm proud of.
