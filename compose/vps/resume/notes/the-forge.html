how the forge runs

july 2026 · the homelab setup, start to finish

hart forge is two sites and a tunnel: a proxmox box at home doing the heavy lifting, one small VPS doing the public-facing work, and tailscale stitching them together. nothing at home is port-forwarded. everything is config in git.

the home side

the proxmox host runs a mix of VMs and LXC containers. the big one is a media VM running jellyfin and the full *arr pipeline: sonarr, radarr, prowlarr, bazarr, the downloaders, plus tdarr for transcodes. a request lands in seerr, gets grabbed, sorted, subtitled, and shows up in jellyfin without me touching anything. storage is a mergerfs pool, so disks can come and go without the layout caring.

around it sit smaller guests, each doing one job: caddy as the LAN reverse proxy (wildcard cert via DNS-01), adguard home for DNS, immich for photos, uptime kuma and a beszel hub for monitoring, homepage as the dashboard, and a forgejo mirror of the git server. LXCs where a container is enough, VMs where it isn't.

there's also a mac mini on the infra VLAN. it runs the discord bot, the scheduled ops scripts, and the agent (more on that below).

the edge

the VPS is small on purpose (4GB of RAM from IONOS) and it only holds what should survive the house losing power: identity, git, notifications, and a handful of apps worth reaching from anywhere. pocket ID does passkey OIDC, tinyauth handles forward auth for apps that can't do OIDC themselves, forgejo is the primary git server (the LAN one is just a mirror), ntfy pushes alerts to my phone, and calibre-web + shelfmark run the library. uptime kuma watches it all from outside.

caddy fronts everything, built with xcaddy so the crowdsec bouncer is compiled in. every vhost imports one shared hardening snippet for logging, crowdsec, security headers, and compression, so a new service can't accidentally skip the baseline. crowdsec decisions also land in nftables, so banned IPs get dropped at the kernel, not just at the proxy.

ingress is cloudflare-only: the firewall accepts 80/443 from cloudflare's published ranges and nothing else, and caddy requires cloudflare's origin-pull client cert on top of that. a weekly job refreshes the ranges in both the firewall and caddy's trusted proxies, because those lists rot. ssh isn't on the internet at all. it listens on tailscale only.

everything as code

ansible playbooks cover the fleet: health checks, updates, config backups, drift detection. docker compose files are per host and per service, pinned tags, no :latest, with diun watching for image updates and posting to ntfy. all of it is versioned in my forgejo, with a curated public snapshot mirrored to github. secrets are vaulted or env-injected and a couple of addresses are swapped for documentation ranges. otherwise it's the real configs.

monitoring that pages

the rule is push over pull where it matters. uptime kuma probes the services, but the important cron jobs also send heartbeats to kuma push monitors, so a script that silently dies pages me instead of just not showing up in a log. beszel does lightweight metrics on every host, ntfy delivers everything to my phone, and a deadman check on the VPS watches the house from outside.

the agent

the newest piece: a persistent claude-based agent (named paimon, because naming things is the fun part) runs on the mac mini with ssh access to the fleet. it does ops work with me (audits, deployments, debugging, config changes) and it keeps its own memory between sessions, so it knows the infrastructure history as well as i do. a good chunk of what's in the public repo was built or is maintained that way, this page included.

what i'd tell past me

  • put identity and git off-site first. everything else can burn and rebuild.
  • one shared hardening snippet beats per-service discipline every time.
  • heartbeats over hope: if a cron job matters, make its silence loud.
  • pin your image tags. :latest is how you get surprise 3am debugging.
  • write it down as code from day one. the second host is when it pays off.
